  二、核心技术决策解释（面试会问）

  为什么用 Qwen 原生 tool calling 而不是纯文本 ReAct？

  纯文本ReAct:
    模型输出 → 正则解析"Action: xxx\nAction Input: xxx" → 容易格式错乱
    7B模型遵循率: ~60-70%，调试痛苦

  Qwen原生tool calling:
    模型输出 → 结构化的<tool_call>标签 → 确定性解析
    7B模型遵循率: ~90%+，Qwen专门训练过这个格式

  两者结合(本项目采用):
    微调数据中同时包含 Thought思考过程 + tool_call结构化调用
    = ReAct的可解释性 + tool calling的稳定性
    简历上写: "基于ReAct架构设计Agent，利用Qwen原生tool calling保证调用稳定性"

  这在面试中是加分项——说明你理解了不同方案的trade-off并做了工程取舍。

  模块 5：Agent 编排（Day 5-6）

  做什么： 用微调后的 Qwen2.5-7B 驱动完整 Agent 循环

  核心架构：

  class CreditRiskAgent:
      def __init__(self, model_path, local_tools, remote_tools, rag_retriever):
          # 加载微调后的Qwen
          self.tokenizer = AutoTokenizer.from_pretrained(model_path)
          self.model = AutoModelForCausalLM.from_pretrained(model_path,
                                                             torch_dtype=torch.bfloat16,
                                                             device_map="auto")
          # 工具注册（来自MCP）
          self.tools = {**local_tools, **remote_tools}
          self.tool_schemas = self._build_tool_schemas()  # Qwen格式的工具描述
          self.rag = rag_retriever

      def run(self, user_query: str, max_turns: int = 5) -> str:
          # 1. RAG预检索，注入相关背景知识到system prompt
          rag_context = self.rag.search(user_query, top_k=3)

          messages = [
              {"role": "system", "content": f"""你是专业的信用风险评估Agent。
  你拥有以下能力：查询用户数据、调用风险模型、检索风控知识库。
  请先思考分析步骤，再调用合适的工具，最后给出完整的风险评估报告。

  参考知识：
  {rag_context}"""},
              {"role": "user", "content": user_query}
          ]

          # 2. 多轮工具调用循环
          for turn in range(max_turns):
              response = self._generate(messages)

              # Qwen原生tool_call解析 — 结构化的，不需要正则
              if response.tool_calls:
                  for tool_call in response.tool_calls:
                      tool_name = tool_call.function.name
                      tool_args = json.loads(tool_call.function.arguments)

                      # 通过MCP调用工具
                      result = self.tools[tool_name](**tool_args)

                      messages.append({"role": "assistant", "content": response.text,
                                       "tool_calls": response.tool_calls})
                      messages.append({"role": "tool", "content": str(result)})
              else:
                  # 没有工具调用 = 最终回答
                  return response.text

          return "分析轮次已达上限，请简化问题。"

  十、关键成功因素

  1. 微调数据中的tool call格式是生死线 —— 类型2数据（工具调用）必须严格符合Qwen的tool_call
  JSON格式，一条格式错误的数据可能毒害整个训练。生成后务必写脚本校验每条数据的JSON合法性。
  2. 先跑通最小闭环再扩展 —— Day 6
  串联时，先让Agent只用1个工具（比如predict_risk_score）完成一个最简Case。跑通后再逐步加工具。不要一上来就4个工具全串。
  3. Kaggle模型你必须理解 —— 面试官大概率会追问"你的ML模型用了什么特征""为什么选这个模型"。虽然是复现开源代码，但必须搞
  懂里面的特征工程和模型选择逻辑。
  4. 评测必须有微调前vs后的对比 —— 这是简历上最有说服力的数字。用完全相同的50条测试集+相同的Judge
  prompt，分别跑原版和微调版。
  5. 晚上训练前确认脚本能跑 —— 每次晚上挂训练前，先跑10步确认不报错。不要挂了一晚上第二天发现第3步就OOM了。
