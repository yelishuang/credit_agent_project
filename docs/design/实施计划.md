# CreditAgent 实施计划

## 已完成

- ML 风控模型：LightGBM 5 折集成，输入 case_id 输出违约概率 + 风险等级 + top 因素
- RAG 知识库：4 篇风控文档 → BGE-m3 向量化 → FAISS 索引，检索评测通过（Hit@3=85.6%, MRR=76.1%）
- SFT 多轮对话数据：5 类共 2377 条，覆盖审批、查询、知识问答、风险解释、拒绝边界
- 项目目录重构完成，所有路径已修复验证

## 阶段一：SFT 数据遴选与格式化

目标：从 2377 条原始数据中筛选出高质量子集，转为 Qwen2.5 微调可用的格式。

具体工作：
- 编写数据校验脚本，检查每条数据的 JSON 合法性、角色交替是否正确、tool_call 参数是否符合 schema
- 按质量过滤：去掉格式错误、内容重复、逻辑不通的样本
- 按比例采样：保持 5 类数据的合理分布（A 审批类占大头，E 拒绝类占少量）
- 输出为 Qwen2.5 chat template 兼容的 JSONL 格式
- 划分 train/val（建议 90%/10%）

完成标准：得到 ~1500 条干净的训练集 + ~150 条验证集，格式能被 transformers tokenizer 正确解析

## 阶段二：LoRA 微调 Qwen2.5-14B-Instruct

目标：在 A6000 上完成 LoRA SFT，让模型学会信贷领域的工具调用和专业分析。

具体工作：
- 编写训练脚本，基于 transformers + peft + trl
- LoRA 配置：r=16, alpha=32, target_modules 覆盖 q/k/v/o/gate/up/down_proj
- 训练参数：3 epochs, batch_size=4, gradient_accumulation=4, lr=2e-4, bf16, cosine scheduler
- 先跑 10 步 sanity check 确认不 OOM、loss 在下降
- 正式训练（预计 2-3 小时/轮），晚上挂着跑 2-3 组超参对比（调 r=8 vs 16, lr=1e-4 vs 2e-4）
- 训练完成后合并 LoRA 权重，保存完整模型

完成标准：训练 loss 收敛，合并后的模型能正常加载并生成文本

## 阶段三：微调模型验证

目标：确认微调后的模型能正确进行工具调用和专业分析。

具体工作：
- 加载微调后模型，手动测试 10-20 个 case
- 重点验证：是否正确输出 tool_call 格式、思考链是否合理、最终报告是否专业
- 对比不同超参组的输出质量，选出最优 checkpoint
- 如果工具调用格式遵循率低于 80%，分析原因（数据质量？格式问题？训练不足？），调整后重训

完成标准：工具调用格式遵循率 > 85%，能完成「查数据→跑模型→查政策→出报告」的完整链路

## 阶段四：MCP 工具服务实现

目标：将数据查询和模型预测封装为 MCP 工具，供 Agent 调用。

具体工作：
- 实现 MCP Local Server（stdio 通信），注册两个工具：
  - query_user_credit_data：输入 user_id，从 Kaggle 测试集 parquet 中查询客户特征
  - predict_risk_score：输入特征字典，调用 LightGBM 模型返回违约概率和风险等级
- 实现 search_knowledge_base 工具：调用已有的 FAISS 检索函数
- 编写工具测试：每个工具单独调用验证输入输出

完成标准：三个工具均可通过 MCP 协议正常调用，输入输出格式与 SFT 训练数据中的 tool_call/tool_response 一致

## 阶段五：Agent 编排

目标：用微调后的 Qwen2.5-14B 驱动完整的 Agent 循环。

具体工作：
- 编写 Agent 主循环：接收用户输入 → 模型生成（可能包含 tool_call）→ 解析并执行工具 → 将结果回填 → 继续生成，直到模型输出最终回答
- 实现 Qwen2.5 原生 tool_call 格式的解析器
- RAG 预检索：用户提问时先检索相关知识，注入 system prompt
- 设置最大轮次限制（5 轮），防止死循环
- 实现命令行交互界面

完成标准：在命令行输入「帮我审批客户 ID=57543 的贷款申请」，Agent 能自动完成多轮工具调用并输出完整审批报告

## 阶段六：LLM-as-a-Judge 评测

目标：量化微调效果，产出简历可用的对比数据。

具体工作：
- 准备 50 条测试用例（覆盖审批、查询、知识、解释、拒绝 5 类场景）
- 分别用微调前（原版 Qwen2.5-14B-Instruct）和微调后模型生成回答
- 用 Claude API 作为 Judge，从 4 个维度打分：事实准确性、专业性、完整性、可操作性（1-5 分）
- 统计微调前后各维度均分，计算提升百分比
- 输出评测报告和对比表

完成标准：评测流程跑通，微调后模型综合评分显著高于微调前基线

## 阶段七：收尾

目标：代码整理，确保项目可复现。

具体工作：
- 补充 requirements.txt
- 清理无用文件和临时输出
- 确保从零开始能跑通：安装依赖 → 构建 RAG → 加载模型 → 启动 Agent
