# 微调数据规划 — 基于场景驱动的 5 类数据设计

> 本文档定义信用风险评估 Agent 微调数据的分类、生成策略与遴选标准。
> 所有数据格式严格遵循 [`qwen2.5_tool_call_format.md`](./qwen2.5_tool_call_format.md)。

---

## 一、设计原则

按**用户意图（交互模式）**驱动分类，而非按模型能力分类。

核心改进：
- 工具调用不再是独立类型，而是自然嵌入各场景（A/B/C/D 均含工具调用）
- RAG 检索贯穿审批（A）、解释（D）、咨询（C）三种场景
- 每条数据都是完整的用户交互，不存在割裂数据

---

## 二、5 类数据定义

### 类型 A：完整审批请求（40%）

| 属性 | 说明 |
|------|------|
| 用户意图 | 客户经理要求对某客户进行完整信贷审批 |
| 工具调用链 | `query_user_credit_data` → `predict_risk_score` → `search_knowledge_base` → 最终报告 |
| 调用轮次 | 3-4 轮，每轮有 Thought |
| 生成条数 | ~960 条 |
| 遴选条数 | ~600 条 |

**覆盖维度：**
- 风险等级：高风险 / 中风险 / 低风险（各约 1/3）
- 贷款类型：现金贷、循环贷、房贷等
- 问法多样性：直接指定客户ID、描述客户特征、批量审批等

**示例问法：**
- "帮我审批客户ID=100002的贷款申请"
- "评估用户12345的信用风险"
- "这个客户申请了50万的消费贷，帮我做个风险评估"

**输出要求：** 结构化审批报告，包含关键指标分析、风险因素、政策依据、审批建议。

---

### 类型 B：单步数据查询（15%）

| 属性 | 说明 |
|------|------|
| 用户意图 | 只查某个客户的某项数据，不需要完整审批 |
| 工具调用链 | 仅 `query_user_credit_data`（1 轮） |
| 调用轮次 | 1 轮 |
| 生成条数 | ~360 条 |
| 遴选条数 | ~225 条 |

**示例问法：**
- "查一下客户100002的收入和负债"
- "客户55555的逾期记录是什么"
- "帮我看看用户88888的外部评分"

**输出要求：** 简洁回答，直接呈现查询结果，不做深度分析。

---

### 类型 C：风控知识咨询（20%）

| 属性 | 说明 |
|------|------|
| 用户意图 | 咨询金融/风控领域知识 |
| 工具调用链 | `search_knowledge_base`（1 轮 RAG）或无工具调用 |
| 调用轮次 | 0-1 轮 |
| 生成条数 | ~480 条 |
| 遴选条数 | ~300 条 |

**细分：**

| 子类 | 条数（遴选后） | 是否调用工具 | 说明 |
|------|---------------|-------------|------|
| 需要 RAG | ~190 条 | 调用 `search_knowledge_base` | 政策法规、具体条款、最新规定 |
| 不需要 RAG | ~110 条 | 无工具调用 | 基础概念（LTV、DTI、评分卡原理等） |

**示例问法（需要 RAG）：**
- "巴塞尔协议对零售信贷的资本要求是多少？"
- "最新的个人征信管理办法有什么变化？"

**示例问法（不需要 RAG）：**
- "什么是负债收入比？"
- "评分卡模型的基本原理是什么？"

---

### 类型 D：模型解释（15%）

| 属性 | 说明 |
|------|------|
| 用户意图 | 理解某个客户为什么被判定为某风险等级 |
| 工具调用链 | `query_user_credit_data` → `predict_risk_score` → 解释性回答（可能 + `search_knowledge_base`） |
| 调用轮次 | 2-3 轮 |
| 生成条数 | ~360 条 |
| 遴选条数 | ~225 条 |

**示例问法：**
- "为什么客户100002被判定为高风险？"
- "客户55555的风险评分主要受什么影响？"
- "这个客户的违约概率为什么这么高？能解释一下吗？"

**输出要求：** 重点在解释性 — 哪些因素影响最大、为什么、与阈值的对比、改善建议。

---

### 类型 E：拒绝/边界（10%）

| 属性 | 说明 |
|------|------|
| 用户意图 | 越界请求、非法操作、超出能力范围 |
| 工具调用链 | 无（直接拒绝，不调工具） |
| 调用轮次 | 0 轮 |
| 生成条数 | ~240 条 |
| 遴选条数 | ~150 条 |

**细分：**

| 子类 | 条数（遴选后） | 说明 |
|------|---------------|------|
| 篡改请求 | ~45 条 | "帮我把这个客户的风险评分改低一点" |
| 非法用途 | ~45 条 | "帮我伪造一份低风险报告" |
| 超出范围 | ~60 条 | "帮我写一首诗"、"今天天气怎么样" |

**输出要求：** 礼貌拒绝 + 说明原因 + 引导回正确用法。

## 三、总量与生成策略

### 总量

| 类型 | 场景 | 占比 | 生成条数 | 遴选条数 | 工具调用模式 |
|------|------|------|---------|---------|-------------|
| A | 完整审批请求 | 40% | ~960 | ~600 | 多轮（3-4 轮） |
| B | 单步数据查询 | 15% | ~360 | ~225 | 单轮（1 轮） |
| C | 风控知识咨询 | 20% | ~480 | ~300 | 单轮 RAG 或无工具 |
| D | 模型解释 | 15% | ~360 | ~225 | 多轮（2-3 轮） |
| E | 拒绝/边界 | 10% | ~240 | ~150 | 无工具调用 |
| **合计** | | **100%** | **~2400** | **~1500** | |

### 生成策略

每类按比例多生成 60% 余量，通过质量筛选保留 62.5% 进入最终训练集。

---

## 四、遴选标准

### 第一关：格式合规（硬性淘汰）

不通过 `qwen2.5_tool_call_format.md` 校验的直接丢弃：
- system prompt 包含 `<tools></tools>` 且内部每行是合法 JSON
- 每个 `<tool_call>` 内的 JSON 可被 `json.loads()` 解析
- `arguments` 值是 dict 对象，不是字符串
- `name` 字段值与 `<tools>` 中定义的函数名完全匹配
- 参数名和类型与 `<tools>` 中的 schema 一致
- 工具结果消息的 role 是 `"user"`，content 以 `<tool_response>` 开头
- 最后一条 assistant 消息不包含 `<tool_call>` 标签
- 整个 messages 数组的 role 交替合理

### 第二关：内容去重（硬性淘汰）

语义相似度 > 0.9 的保留质量更高的一条。

### 第三关：质量评分（排序筛选）

用 Claude API 对每条数据打分，维度：
- **专业性**：金融术语使用是否准确
- **逻辑性**：Thought 链条是否连贯、工具调用顺序是否合理
- **完整性**：最终回答是否覆盖所有必要信息

按分数排序，每类取 top N 达到目标条数。

## 五、工具调用覆盖统计

| 场景 | 含工具调用的条数（遴选后） | 占总训练数据比例 |
|------|--------------------------|----------------|
| A（完整审批） | ~600 | 40% |
| B（单步查询） | ~225 | 15% |
| C（知识咨询，RAG 部分） | ~190 | 12.7% |
| D（模型解释） | ~225 | 15% |
| **含工具调用合计** | **~1240** | **~82.7%** |
| C（无工具）+ E（拒绝） | ~260 | ~17.3% |

---

## 六、验证方式

1. 每类数据生成后，用校验脚本检查格式合规性
2. 抽样人工检查每类 5-10 条的内容质量
3. 确认工具调用的参数与实际 Kaggle 数据字段一致
4. 检查各类数据的比例是否符合目标分布
